# Claude Code 개발 워크플로우 티어

Claude Code로 개발 작업을 할 때 작업 규모에 따라 워크플로우 티어를 선택하는 기준.
핵심 원칙: 필요한 것보다 무거운 티어를 쓰지 않는다. 낮은 티어로 시작하고 복잡도가 드러나면 올린다.

## 티어 정의

### Tier 0 — Direct (에이전트 없이 직접)
- **조건**: 단일 파일, 10줄 미만, 명확한 수정 또는 설정 변경
- **액션**: 직접 파일 편집 → `github` 스킬로 커밋
- **예시**: 오타 수정, 변수명 변경, 버전 번프, config 수정

### Tier 1 — Plan + Implement
- **조건**: 1–3개 파일, 명확한 범위, 병렬 처리 불필요
- **액션**: Plan 모드 → 구현 → `github` 스킬
- **예시**: 버그 수정 (원인 파악됨), 소규모 신기능

### Tier 2 — Plan + Parallel Agents
- **조건**: 3–10개 파일, 또는 독립적인 여러 관심사
- **액션**: Plan 모드 → 1–3개 개발자 에이전트 (병렬, worktree 격리) → 빌드 → `github` 스킬
- **예시**: 여러 파일에 걸친 리팩토링, 독립 모듈이 있는 신기능

### Tier 3 — Full Workflow
- **조건**: 10개 이상 파일, 아키텍처 영향, 또는 횡단 관심사
- **액션**: `/workflow` 스킬 전체 실행 (QA, 리뷰어, 빌드 루프 포함)
- **예시**: 대규모 신기능, 아키텍처 변경, 다중 서브시스템

## 라우팅 시그널

| 시그널 | 티어 |
|--------|------|
| "오타", "이름 변경", "버전 번프", config 수정 | 0 |
| 버그 수정 (원인 이미 파악됨) | 0–1 |
| 소규모 신기능, 단일 관심사 | 1–2 |
| 여러 파일에 걸친 리팩토링 | 2–3 |
| 대규모 신기능, 여러 서브시스템 | 3 |
| "워크플로우 실행" 명시적 요청 | 3 |

## 에스컬레이션 원칙

구현 중 숨겨진 복잡도가 드러나면 올린다. 반대로 처음 예상보다 간단하면 내린다.
의심스러우면 낮은 티어로 시작한다 — 과소 설계 후 에스컬레이션이 과대 설계보다 낫다.

## Workflow Stages (Tier 3 전체 파이프라인)

```
[0] 플랜 doc 작성
[1] 기획 (opus 모델)
[2] 아키텍트 에이전트
[3] 개발자 에이전트 ×N (병렬)
[4] QA 작성자 ×N (병렬)
[5] 리뷰어 ×N
[6] QA 실행자 ×N
[7] 빌드 & 테스트 (실패 시 최대 3회 반복)
[8] docs/specs 업데이트
[9] 커밋 → PR
```

빌드 커맨드는 에이전트 내부에서 실행하지 않는다 — 항상 외부에서.
[7] 실패 = 미완료 작업 (통과할 때까지 루프).

## Completion 명령 (wrap up / commit / finalize)

"마무리", "커밋", "wrap up" 요청 시 순서:

1. `docs/exec-plans/active/` → `completed/`로 완료된 플랜 이동
2. 설계 결정이 바뀌었으면 `docs/design-docs/` 업데이트
3. 일반화할 수 있는 지식 캡처 (memory 또는 skill 파일에 기록)
4. `github` 스킬로 커밋 — 영어 메시지 + Co-Authored-By
5. PR 생성
