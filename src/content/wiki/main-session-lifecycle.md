## 핵심 원칙

**메인세션 = 티켓 하나 = 요구사항부터 리뷰 반영 완료까지**

- Context가 차오르면 AI가 서둘러 마무리한다 → 결과물 품질 저하
- 메인세션 context는 **이번 작업의 원재료**(요구사항/설계/구현)에만 쓴다
- 문서화·평가기준추출·리뷰는 모두 **메인세션 밖**에서 수행

## 라이프사이클 단계

| 단계 | 세션 | 역할 |
|------|------|------|
| 1. 요구사항 + 설계 | 메인세션 | 원재료 생성 |
| 2. 설계의도 + ADR 기록 | **Fork 세션** (폐기) | 설계 당시 맥락 보존 |
| 3. 평가기준 추출 | **Sub-agent** (별도 세션) | `code-quality-guide.md` 생성 |
| 4. 구현 | 메인세션 + guide | 코드 작성 |
| 5. 리뷰 | **별도 리뷰 agent** (새 세션) + guide | 리포트 생성 |
| 6. 리뷰 반영 | 메인세션 | 피드백 적용 → 티켓 종료 |

## Fork를 쓰는 이유 (문서 전달 방식 대비)

| 방식 | 장점 | 단점 |
|------|------|------|
| 같은 메인세션이 문서 작성 | 간단 | Context 오염 — 구현에 쓸 공간 갉아먹음 |
| 문서만 넘겨 새 세션 | 깔끔 분리 | 암묵적 맥락 손실 (버린 대안, 네이밍 이유 등) |
| **Fork → 폐기** | 설계 순간의 맥락 그대로 보존 | 플랫폼 지원 필요 |

**원칙**: Fork는 "그 시점의 설계 의도를 가장 가깝게 담는" 방법. 문서 전달은 사람 검수가 필요하고 그래도 완벽하지 않다.

## Sub-agent로 평가 기준을 뽑는 이유

- `code-convention.md` + `adr.md`은 프로젝트가 오래될수록 누적되어 커진다
- 전체를 메인세션에 넣으면 이번 티켓과 무관한 규칙이 대부분 → context 낭비
- Sub-agent가 관련 부분만 추려 `code-quality-guide.md`로 제공

## `code-quality-guide.md`의 이중 사용

1. **구현 시**: 메인세션이 규칙으로 참조
2. **리뷰 시**: 리뷰 agent가 판정 기준으로 사용

→ 같은 기준으로 구현하고 같은 기준으로 검증. 일관성 보장.

## 리뷰 기준 두 문서

| 문서 | 역할 | 성격 |
|------|------|------|
| `code-convention.md` | 코딩 스타일 지침 | 상대적으로 안정. 언어마다 여러 개 가능 |
| `adr.md` | 기술적 결정 기록 | 누적형. 결과만이 아니라 **고민한 대안 + 선택 이유**도 담김 |

- 이 둘이 잘 쌓여 있으면 AI 리뷰와 사람 리뷰 구별 어려움
- ADR이 부실하면 → 보안·중요 영역에서 사람 리뷰 필요

## 사람의 역할 재배치

| 시점 | 사람 | AI |
|------|------|-----|
| 작업 전 | 설계의도 문서 검토, 평가기준 검토 | — |
| 구현 중 | — | 메인세션 |
| 리뷰 | (중요/보안만) | 별도 리뷰 agent |
| 리뷰 반영 | — | 메인세션 |

**결과**: 코드 리뷰 병목 해소. 사람은 **더 앞단**(의도/기준)에 시간을 씀.

## 세션 분리 효과

- 리뷰 agent는 메인세션과 context가 분리됨 → "남이 읽는 것과 비슷한" 시각
- 메인세션이 자기 코드를 리뷰하면 blind spot 남김 → 별도 세션 필수

## 리뷰 반영을 메인세션이 하는 이유

- 구현한 세션이 "왜 그렇게 짰는지"를 가장 잘 안다
- 피드백 해석과 수정이 원 설계 의도 안에서 일관되게 이뤄짐
- 따라서 메인세션은 **리뷰 반영 완료까지** 유지해야 함

## 전제 조건

- ADR을 계속 누적하는 규율 — 결정만 아니라 **고민한 alternatives**도 기록
- Fork 가능한 플랫폼 지원 (Claude Code의 세션 fork 기능 등)
- Sub-agent 호출로 메인세션에 결과만 반환받는 패턴
